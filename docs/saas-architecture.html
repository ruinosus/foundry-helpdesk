Foundry Assured
como SaaS multi-tenant

O mesmo concierge, agora um SaaS híbrido: nós orquestramos (control plane), mas dados, compute e credenciais ficam na nuvem do cliente (BYO). Um só código, três modos de entrega — self_hosted, dedicated, shared — e quatro domínios. Tudo seguindo a orientação Microsoft, registrado em 11 ADRs.

Arquitetura SaaS01 / 08
A ideia central

Nós orquestramos · os dados ficam com o cliente

Operamos o control plane (UI + orquestrador + RBAC + gates de assurance). Todos os dados, compute, segredos e conexões vivem no data plane do cliente. O store do control plane guarda só config + metadados de conexão por tenant — nunca segredo, nunca dado do cliente.

nós operamos

Control plane

Next.js + FastAPI + RBAC + assurance. Resolve o tenant pelo tid do token, carrega a config daquele tenant, gera um token brokered (OBO / passthrough) e chama o data plane do cliente. Guarda apenas referências.

na nuvem do cliente

Data plane (BYO)

O Foundry project do cliente é a fronteira natural de isolamento: KB / Azure AI Search, memory store, storage, conexões. O agente chama o Foundry deles com o token brokered — o isolamento é, em boa parte, automático.

🔒 Único choke point de tenant-scoping (auditável) · tudo fail-closed: tenant não onboarded → erro claro, nunca cai em outro tenant.

Um código · três modos de entrega

self_hosted  ·  dedicated  ·  shared

O mesmo código roda em três configurações, variando só onde o control plane roda e quantos tenants ele serve. O único ponto de variação é a seam DEPLOYMENT_MODE + TenantConfigProvider (Single ⇄ Multi). O resto é idêntico.

Isolamento é uma escolha de deployment, não um rewrite. O produto self-hosted de hoje vira uma variante de stamp — sem fork. O dedicated é o self-hosted que nós operamos; o shared é o multi-tenant de verdade. (ADR-001 · ADR-007)
Três modos03 / 08
Deployment Stamps · os veículos

Managed Application + Azure Lighthouse

dedicated

Managed Application

Publicamos o control plane como Managed App: ele instala num resource group da assinatura do cliente que o cliente não modifica diretamente — e nós operamos. "Tudo é do cliente", inclusive o control plane, com a gente como operador.

Artefato: Bicep que compõe os módulos infra/ existentes → mainTemplate.json + createUiDefinition.json. Re-parametrização, não duplicata.

shared

Azure Lighthouse

O cliente delega escopos específicos à nossa tenant; gerenciamos os recursos do data plane cross-tenant sem nunca possuir os dados. Least-privilege (Reader + Monitoring Contributor + Log Analytics Reader — sem Owner/Contributor), revogável e auditável a qualquer momento.

ambos marketplace-publishable fronteiras de propriedade claras validado offline: bicep build + ARM-TTK + CreateUiDefinition sandbox publish + delegação real = infra-gated (runbook)

Stamps · veículos04 / 08
A evolução · todos os sub-projetos construídos

A → B → C → D

Quatro sub-projetos, cada um com uma seam clara e um mecanismo sancionado pela Microsoft — é isso que torna o alvo viável, não invenção.

A

Fundação multi-tenant

A espinha: seam TenantConfigProvider (Single/Multi) + DEPLOYMENT_MODE, Entra multi-tenant, resolução por tid + OBO, memória com namespace por tenant, store de tenant trocável. De-risk: SingleTenant primeiro = zero mudança de comportamento.

ADR-003 · 006 · 007
B

Conexões

Records de Connection por tenant que referenciam conexões do Foundry (nunca guardam segredo) + API admin /tenant e a página de Conexões. Store = Table Storage agora, Azure App Configuration em produção.

ADR-005 · 008
C

Brokering + governança de escrita

As tools MCP do agente de plataforma dirigidas pelas Connections do tenant; credenciais resolvidas de forma nativa Microsoft (OBO p/ audience Microsoft, Foundry connections senão). RBAC por tool; tools de escrita com tool-approval nativo (Approver/Admin).

ADR-009
D

Shared-mode + packaging

Domínios montam globalmente + gate por tenant; DomainAssignment = entitlement por licença. O hosted agent de plataforma (Invocations + Toolbox) e o stamp dedicated (Managed App + Lighthouse); tier→domínios.

ADR-010 · 011
🔁 Convergência: dedicated/self-hosted ponta-a-ponta = A→B→C. Shared ponta-a-ponta = C ∩ D (o OBO multi-tenant de C brokeia sobre a assinatura delegada por Lighthouse que D provisiona).

Identidade & credenciais · "nunca guardamos um segredo"

O control plane resolve a credencial · nunca a lê

Onboarding uma vez por tenant: o admin do cliente faz admin-consent do app multi-tenant (cria um service principal na tenant dele) e delega os recursos do data plane (Lighthouse). Guardamos config + referências de conexão, só isso.

Os quatro domínios · um registry, uma rota genérica

helpdesk · cockpit · selfwiki · platform

Os 4 domínios07 / 08
O domínio platform, hosted · per-tenant

O agente hosted resolve tools por tenant — sem tocar credencial

O twin /platform-hosted é o agente de plataforma empacotado como hosted agent do Foundry. Carrega write-approval HITL, então usa o protocolo Invocations (não Responses) — o stream AG-UI atravessa o Foundry intacto, e o interrupt de aprovação sobrevive ponta-a-ponta.

📦

Container protocol-only

InvocationsHostServer serve o mesmo agente AG-UI do /platform. Tool-auth não é trabalho do container.

🧰

Foundry Toolbox

As tools resolvem por um Toolbox MCP provisionado no project — o broker central de auth. É onde aterrissa o build_hosted_from_connections de C, como config, não código.

🔑

OAuth passthrough

Identidade por tenant via OBO: as tools rodam como o usuário logado, consentimento na primeira vez. A definição do agente só guarda referências.

🚧 O hosted agent + o stamp são authored / infra-gated — o modelo de entrega desenhado e deployável via o D-PACKAGING-RUNBOOK.md (Partner Center / Lighthouse / Toolbox são contra infra real, fora do CI). · ADR-011.
