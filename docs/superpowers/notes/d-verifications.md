# Sub-project D — runtime / SDK verifications (plan Task 0)

Verified against the installed versions on `feature/saas-d-runtime` (`agent_framework` 1.9.0, `agent_framework_ag_ui` co-installed, `azure-ai-projects` 2.2.0). These resolve the two unknowns the D plan gated on (design Open Questions #1 Invocations bridge contract and #2 `require_domain` ordering).

| Unknown | Finding | Impact on the plan |
|---|---|---|
| **Gate dependency ordering** (Open Q#2): does `require_domain` reading `_current_tenant` need `require_user` to have run first via `dependencies=` list position? | `require_user` is importable from `app.core.auth` and, in the **shared** branch, calls `resolve_tenant(user, _tenant_store)` which sets `_current_tenant` (`app/core/auth.py:119-121`). FastAPI resolves the dependency **graph**, not list position, and **caches** a dependency per request — so declaring `require_user` as a sub-dependency (`Depends(require_user)`) inside `require_domain`'s signature guarantees it runs first and exactly once, even though `auth_dependencies()` (`auth.py:138`) also lists `Depends(require_user)` on the endpoint. | **DECISION LOCKED:** `require_domain` declares `require_user` as a FastAPI **sub-dependency** (`Depends(require_user)` in its inner signature). The graph guarantees ordering + `_current_tenant` is set before the gate reads it; FastAPI's cache means it runs once. **No reliance on `add_agent_framework_fastapi_endpoint` honoring `dependencies=` ordering is needed** — Task 6 does not depend on list-position ordering. |
| **Invocations endpoint URL shape** (Open Q#1): is `{project_endpoint}/agents/{name}/endpoint/protocols/invocations` real? | The endpoint **template** `{endpoint}/agents/{agent_name}/endpoint/protocols/{protocol}` is **confirmed** in installed `azure-ai-projects` 2.2.0 (`_patch.py:51`, `:225`; `aio/_patch.py:150`) — but the **only protocol literal present in the SDK is `protocols/openai`** (used by `AIProjectClient.get_openai_client(agent_name=..., allow_preview=True)`). There is **no `protocols/invocations` literal anywhere** in the installed `azure-ai-projects`, `agent_framework`, or `agent_framework_ag_ui`. | The host/path **template** for hosted agents is verified and the `/invocations` shape is consistent with it (by analogy to `/openai`), but the `invocations` protocol itself is **not in any installed library**. The `{name}` and `allow_preview=True` requirements carry over (preview-gated). Task 6 must NOT assume the SDK builds this URL for it — there is no first-party helper. |
| **First-party Invocations client offline** (Open Q#1) | **Does NOT exist offline.** Probing `agent_framework` and `agent_framework_ag_ui` for `invoc`/`host` submodules returned `[]`. `agent_framework_ag_ui` exposes the **server/adapter** side (`AgentFrameworkAgent`, `add_agent_framework_fastapi_endpoint`, `AGUIHttpService`, `AGUIChatClient`) — i.e. how to *serve* AG-UI, not how to *consume* a hosted agent's Invocations endpoint. No Invocations protocol client / SSE parser ships in any installed package. | **No first-party Invocations client to wrap.** The bridge cannot be built on a verified SDK client offline. |
| **Invocations SSE request/response envelope** (Open Q#1) | **NOT determinable offline.** No envelope schema, no SSE event-type model, no request/response shape for the Invocations protocol exists in the installed libraries. Confirming it would require the deployed agent's live endpoint or Microsoft Learn protocol docs — neither pins a concrete, verifiable Python signature here (project rule #1: do not invent the envelope). | **Per plan step 0's explicit allowance:** the `/platform-hosted` bridge (`app/services/hosted.py`) **stays a clean-error skeleton** — route + twin/toggle wired, but raw-SSE streaming returns a clean "not yet available" error rather than a fabricated envelope. **Real streaming lands with the deployed agent (D-packaging / infra-gated E2E)**, where the live endpoint reveals the envelope. Task 6 implements **only** the verified surface (route, toggle, clean error), not a guessed SSE contract. |

**Net:** Open Q#2 is fully resolved and the design decision is locked (sub-dependency, not list ordering). Open Q#1 is **half-resolved**: the agent-endpoint URL *template* is verified (`/protocols/{protocol}`, preview-gated, by-name), but the **`invocations` protocol — its client and its SSE envelope — is not present in any installed library and is not determinable offline**. Per the plan's stopgap-honesty clause, the Invocations bridge ships as a clean-error skeleton in D-runtime; the real SSE wiring is deferred to D-packaging against the live hosted agent.
