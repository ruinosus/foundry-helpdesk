# Sub-project D-packaging — #3199 write-approval-over-AG-UI status + live-verify procedure (plan Task 9)

Verified on `feature/saas-d-packaging` against the installed backend venv (`apps/backend/.venv`, Python 3.12): `agent-framework` **1.9.0** (core 1.9.0), `agent-framework-ag-ui` **1.0.0rc5**.

**Question:** Does native MCP `approval_mode=always_require_approval` (a.k.a. `"always_require"`) actually EXECUTE the approved write over the **AG-UI adapter** in this version? Historically it did NOT (agent-framework **#3199**) — which is why the live `/platform` build uses `approval_mode="never_require"` and gates writes by role-visibility instead (`app/agents/mcp/tools.py:6-7,96`), while the hosted-path builder (`build_hosted_from_connections`, `tools.py:143-144`) DOES set `always_require_approval` for writes, pending verification.

## Status: #3199 is FIXED in the installed surface — NATIVE, no middleware needed.

| Unknown | Finding | Decision |
|---|---|---|
| **Is #3199 open or fixed? Which release?** | **CLOSED / Done.** Issue [microsoft/agent-framework#3199](https://github.com/microsoft/agent-framework/issues/3199) ("MCP Tools with `approval_mode` set to `Always_requires` are not executed when using AG-UI") was filed 2026-01-13 against `1.0.0b260107` and fixed by PR [#3212](https://github.com/microsoft/agent-framework/pull/3212) (merged 2026-01-15). Root cause: the AG-UI orchestrator's `collect_server_tools()` didn't include MCP tools attached to the agent, so an approved MCP call failed with "function not found". The fix added a public `mcp_tools` attribute to the agent and updated `collect_server_tools()` to include MCP tool functions in the approval-execution path. | Our `agent-framework 1.9.0` / `agent-framework-ag-ui 1.0.0rc5` are the stable line that succeeds the `1.0.0b2601xx` betas the bug lived in — well past the 2026-01-15 merge. |
| **Is the fix actually PRESENT in the installed code (not just by date)?** | **YES — confirmed in the installed bytes (per rule #1, not assumed from a version number).** (1) The public attribute PR #3212 added is present: `app_framework/_agents.py:749` — `self.mcp_tools: list[MCPTool] = [tool for tool in normalized_tools if isinstance(tool, MCPTool)]`. (2) The adapter's approval-execution path reads it: `agent_framework_ag_ui/_orchestration/_tooling.py` defines `collect_server_tools(agent)` which does `mcp_tools = getattr(agent, "mcp_tools", None)` and, when present, merges `_collect_mcp_tool_functions(mcp_tools)` into the returned tool set (`_tooling.py:35-62`), alongside the `_try_execute_function_calls` approval path in `_agent_run.py`/`_tooling.py`. This is exactly the PR #3212 change ("`collect_server_tools()` now includes MCP tool functions for approval execution"). | The native `always_require_approval` interrupt-and-resume now has the tool visible to execute on resume. **No `RequestInfoEvent` middleware fallback is required** (ADR-009's stated fallback is NOT built). |

**Net:** native MCP write-approval over AG-UI is wired in this version. The conditional middleware in plan Task 9 Step 3 is **NOT implemented** — the precondition ("#3199 unfixed") is false. The code already reflects the intended design: the hosted-path builder sets `always_require_approval` for writes (`tools.py:143-144`), relying on this native interrupt; the live `/platform` builder keeps `never_require` + role-visibility gating (`tools.py:96`) as defense-in-depth (a Reader never sees a write tool at all). The frontend `TicketApproval.tsx` already taps the `request_info` CUSTOM event and resumes via `runAgent({ resume: [...] })` for both the helpdesk ticket HITL and the native MCP tool-approval shape.

## Live-verification procedure (INFRA-GATED — needs a running stack + deployed agent)

Offline confirms the fix is *present in the SDK*. What it does NOT confirm is the *exact AG-UI wire shape* of the native MCP `ToolApprovalRequestContent` interrupt as it surfaces through the Foundry-hosted path — i.e. whether `TicketApproval.tsx`'s discriminator (`tool_name ?? name ?? function_name ?? toolName` / `arguments ?? args ?? …`, `TicketApproval.tsx:72-83`) matches the real payload, and whether resume re-executes the write. That is the remaining live gate.

**Preconditions (not runnable in CI):** the shared stack up (backend `app.main:app` + frontend `npm run dev`); the platform domain configured + a write-capable MCP connection enabled for the caller's tenant with the caller holding **Author/Admin** (so a write tool is visible); for the hosted leg, the deployed `platform-concierge` agent + its Foundry Toolbox (ADR-011) and `TOOLBOX_NAME` wired.

**Procedure — run twice, against `/platform` (live, in-process build) AND `/platform-hosted` (the deployed Invocations agent via `stream_platform_agui`):**

1. In the Assurance Console, select the platform domain. As an **Author/Admin**, prompt the concierge to perform a **write** through an MCP server whose write tool is configured `always_require_approval` (e.g. create/comment on an item via the GitHub/ADO MCP server).
2. **Assert the interrupt surfaces:** the agent emits a `request_info` CUSTOM event; `TicketApproval.tsx`'s `onEvent` tap catches it, discriminates it as `kind: "tool"` (carries a tool name + args, not a `summary`), and renders the approval card. Confirm the rendered tool name + arguments match the requested write. If the discriminator misses (card renders as `kind: "ticket"` or not at all), adjust the field mapping at `TicketApproval.tsx:80-83` to the real `ToolApprovalRequestContent` field names and note them.
3. **Assert resume executes the write:** click **Approve** → frontend calls `runAgent({ resume: [{ interruptId: id, status: "resolved", payload: true }] })`. Confirm the agent *executes the approved MCP write* (the pre-#3212 bug was a "function not found" here) and streams a normal terminal — the side effect (issue/comment created) actually happens. Then repeat with **Reject** (`payload: false`) → confirm the write is NOT performed and the run terminates cleanly.
4. **Hosted parity:** repeat 1-3 on `/platform-hosted`. Because the bridge is an AG-UI passthrough (ADR-011 — `stream_platform_agui` relays the SSE bytes 1:1), the same `request_info` interrupt + `runAgent({ resume })` must surface identically through the deployed agent. Confirm the approval card + resume behave the same as the live leg.

**Outcome to record after live-verify:** for each leg — interrupt-surfaces ✅/✗, discriminator-matches (+ any field-name corrections), resume-executes-write ✅/✗. If any leg shows the native approval still does NOT execute the write over AG-UI in the deployed Foundry path (regression / Foundry-hosting-specific gap), THEN — and only then — implement the ADR-009 fallback: a framework `RequestInfoEvent`-emitting middleware that pauses write tools and emits the approval interrupt (NOT a hand-rolled workflow). Offline, that precondition is false, so the middleware is deferred.
