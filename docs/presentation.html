Dois modelos de entrega · o mesmo agente

Live workflow  ⇄  Hosted agent

DimensãoLive workflow (AG-UI)Hosted agent (Foundry)
ProtocoloAG-UI (SSE, streaming)Responses (OpenAI-style)
Passos do workflow na tela✔ sim — triage/retrieve/resolve ao vivo✘ não — só a resposta final
Human-in-the-loop (aprovação)✔ sim — approval card✘ não — request→response
IdentidadeOBO — chama como o usuárioIdentidade própria do agente
Memória por usuário✔ sim✘ não (variante limpa)
Onde rodaNosso FastAPI (nós hospedamos)Foundry gerencia (autoescala, idle=$0)
Quem chamaNosso appQualquer app/time, por nome
O hosted é CÓDIGO num container (seu workflow Python), não um prompt. As respostas diferem um pouco de propósito: ele é uma variante simplificada (sem memória/OBO/HITL) — e LLMs não são determinísticos. A diferença é o ponto: dois jeitos de entregar o mesmo agente.
Modelos de entrega05 / 07
Confiança

Avaliado e rastreável — não “confie em mim”

  • ▰
    Gate de policies (determinístico) — toda resposta cita uma fonte (ou declina) e nunca vaza segredo. Viola → CI falha (exit ≠ 0).
    LocalEvaluator + @evaluator · self-test prova que o gate pega a violação plantada
  • ▰
    Judges do Foundry (LLM-as-judge) — groundedness · relevance · coherence sobre o golden set, com link direto pro relatório no portal.
    FoundryEvals — groundedness 9/9 no último run
  • ▰
    Tracing OpenTelemetry — cada interação vai pro Application Insights; scores ligados aos traces.
🟢 Página /evals no app mostra os runs com os scores e o link pro portal do Foundry
