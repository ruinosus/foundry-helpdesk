O que é

Não é autocomplete. Não é um SaaS. É um pipeline local e auditável.

"Deep-wiki" virou categoria — mas as opções diferem no que importa: onde roda, se cita a fonte, e se você confia no resultado. Esta abordagem foi desenhada para ser verificável.

Este mecanismo

Roda local (uv run …), chamando o modelo do Foundry. Cada afirmação cita um arquivo real; um gate barra o que não resolve; o acesso segue a fonte.

local · gated · citável

A geração não depende do editor: é um script Python (app.knowledge.wiki_builder) que lê o código deterministicamente e usa o modelo só para planejar/escrever/verificar — então roda em CI, cron ou na sua máquina.

Como é feita · geração

Ler a fonte → planejar → escrever → verificar → barrar.

Uma passada por área do monorepo. O modelo nunca "inventa do nada": ele recebe os arquivos reais e é forçado a citar caminhos que existem. Um verificador reescreve cada página removendo o que não tem suporte.

🛡️ Gate de fidelidade — ≥ 80% das citações têm que resolver para um arquivo que existe; senão o bundle é escrito só para inspeção e não é ingerido.
Achado real do dogfood: virar o gate sobre o próprio repo expôs um bug nele (um regex que casava .js dentro de .json), reprovando injustamente as áreas ricas em config. Corrigido — fidelidade subiu de 37%/50% para 94%/98%.

Quem vê o quê · Entra ID + On-Behalf-Of

A resposta é fundamentada — e só no que você pode ler.

Num corpus corporativo (SharePoint, repositórios, ADLS) nem todos veem tudo. O agente consulta a base como você — não como ele mesmo — e a recuperação é aparada pela sua identidade antes do modelo ver qualquer coisa.

🔒 O token do chamador viaja no header x-ms-query-source-authorization; o Azure AI Search resolve os grupos do usuário e apara o resultado — o grounding vira o limite de segurança.
🧭

Acesso segue a fonte

Cada documento carrega os grupos de leitura da sua origem (campo groups) — um time do GitHub, um grupo do ADLS, um grupo do SharePoint. Nomes resolvem para object-IDs do Entra. Nenhuma classificação no código; o acesso é dado, não palpite.

🛡️

Defesa em profundidade

Camada C (serviço): o passthrough da identidade apara no próprio Search quando ele suporta. Camada B (app): uma busca direta como você revela seu direito, e todo chunk fora dele é descartado antes do modelo. Autoritativo — vem da ACL do Search, não de heurística.

🚫

Fail-closed · à prova de injeção

Documento sem grupo resolvível → ninguém vê. E o trim é por identidade, não pelo prompt: jailbreak não amplia acesso, porque a poda acontece independente da pergunta. Red-team medido: ASR 0% em 8 ataques adversariais.

Custo previsto · por modelo

A wiki inteira custou $0.30. Em centavos.

Complexidade × tamanho

O custo segue o tamanho da fonte. A fidelidade, não.

A saída é limitada — ~7 páginas por área, independente do tamanho. O que varia é o custo (proporcional aos tokens de entrada). E a qualidade não cai com o tamanho: infra (4 arquivos) e backend (65) ficam ambos acima de 95%.
